## 1. 项目概述

### 1.1 项目背景

在信息爆炸的时代，个人在日常学习、工作和生活中会持续产生和积累大量的信息碎片——阅读笔记、技术心得、项目文档、生活记录、灵感片段等。这些内容分散在各个平台（本地 Markdown 文件、云笔记、社交媒体、即时通讯记录等），导致以下痛点：

- **信息分散**：知识散落各处，缺乏统一检索入口，查找效率低下
- **格式不一**：不同工具存储格式各异，难以建立结构化的知识体系
- **无法沉淀**：零散的记录难以形成系统化的知识网络，回顾和复用困难
- **归属感弱**：依赖第三方平台，数据归属和隐私控制能力不足

基于以上痛点，zblog 定位为一个集 **个人知识库**、**个人记事本** 与 **个人资料管理** 于一体的综合型个人博客系统：

#### 1.1.1 个人知识库

面向长期知识沉淀场景，将分散的学习笔记、技术文章、读书心得等系统化整理：

- 支持 Markdown 富文本编辑，兼顾可读性与可移植性
- 通过分类体系与标签网络构建知识图谱，实现知识间的双向链接与标签共现发现
- 提供全文搜索能力，快速定位历史积累的任意内容
- 支持文章版本历史，追踪知识的演进过程

#### 1.1.2 个人记事本

面向日常记录与快速写作场景，降低记录门槛：

- 支持快速草稿录入，随手记录灵感与想法
- 提供简洁无干扰的写作体验（类似纯文本编辑器）
- 支持按时间线回溯，像翻阅日记本一样回顾过往记录
- 可对记事内容设置私密/公开状态，灵活控制可见范围

#### 1.1.3 个人资料管理

面向各类参考资料的归档与检索场景：

- 支持文件上传与附件管理（图片、PDF、文档等）
- 提供链接收藏功能，将外部优质资源纳入统一管理
- 通过标签和分类对资料进行多维索引
- 支持批量导入导出，保障数据可迁移性

综上，zblog 并非传统意义上仅用于对外展示的博客，而是一个 **以博客形态承载个人知识全生命周期管理** 的工具平台——既是外部分享的窗口，也是内部知识沉淀的仓库。

### 1.2 项目目标

基于 1.1 所述背景与定位，zblog 项目确立以下核心目标：

| 编号 | 目标 | 说明 |
|------|------|------|
| G1 | **知识统一归集** | 将散落在各处的笔记、文章、资料汇聚至单一平台，提供统一检索入口 |
| G2 | **结构化知识体系** | 通过分类、标签、双向链接构建知识网络，让孤立内容产生关联价值 |
| G3 | **低摩擦写作体验** | 提供 Markdown 编辑器与快速草稿能力，降低记录与发布的心理门槛 |
| G4 | **资料一站式管理** | 整合文件存储、链接收藏、批量导入导出，形成完整的个人资料中心 |
| G5 | **公开与私密灵活控制** | 支持内容发布为公开文章或保留为私密笔记，兼顾分享与隐私 |
| G6 | **数据自主可控** | 数据存储在自有服务器，支持一键导出，不依赖第三方平台 |
| G7 | **轻量易部署** | 保持系统架构简洁，降低安装与运维成本，适合个人长期维护 |
| G8 | **SEO 友好** | 公开内容具备良好的搜索引擎可发现性，方便他人检索与引用 |

### 1.3 目标用户

zblog 的核心用户群体为 **技术开发人员**，具体画像如下：

**主要用户：**

- **后端/前端/全栈开发者**：日常积累技术笔记、踩坑记录、架构设计文档，需要结构化知识库辅助快速回顾与复用
- **DevOps / SRE 工程师**：记录运维脚本、故障排查流程、基础设施配置，需要可全文搜索的私有资料库
- **技术管理者 / 架构师**：撰写技术方案、设计文档、团队规范，需要版本历史追踪内容演进

**次要用户：**

- **技术内容创作者**：通过公开文章分享技术见解，建立个人品牌
- **计算机相关专业学生**：整理学习笔记、项目复盘，构建个人知识体系

**用户特征：**

- 熟悉 Markdown 语法，偏好键盘驱动的编辑体验
- 具备自主部署能力，倾向于数据自托管
- 对知识管理有结构化需求，习惯通过分类和标签组织信息
- 需要代码高亮、技术图表等开发者友好特性

### 2.1 前台展示（读者端）

前台是面向读者/访客的博客展示层，是系统对外呈现的窗口。以下需求均以 **匿名访问者视角** 定义，访问范围仅限 **公开已发布** 的文章和页面。文章状态定义如下：

| 状态 | 含义 | 前台可见 | 后台可见 |
|------|------|:---:|:---:|
| 草稿（Draft） | 尚未完成的内容，未设定发布时间 | ✗ | ✓ |
| 私密发布（Private） | 已正式发布，但仅博主本人可见 | ✗ | ✓ |
| 公开发布（Public） | 已发布，所有访客可见 | ✓ | ✓ |

> **说明**：草稿与私密发布的区别在于——草稿是未完成的工作副本，可能无正式发布时间；私密发布有正式发布时间和固定 URL，但仅博主登录后可见，适合存档个人私密笔记。  
> **安全说明**：私密文章和草稿的前台路由（`/post/{slug}`）在未登录状态下返回 404 而非 403，防止通过 HTTP 状态码泄露私密内容的存在性。

#### 2.1.1 文章列表页（首页）

文章列表页为博客首页，是访客到达后的第一入口。首页采用 **左右双栏布局**——左侧为导航侧栏（分类列表 + 标签列表），右侧为文章卡片列表。

| 编号 | 优先级 | 需求描述 |
|------|--------|----------|
| FR-L01 | P0 | **文章卡片列表**：以时间倒序展示已发布文章列表，每篇文章显示标题、摘要（自动截取正文前 N 字符）、发布时间、分类、标签、阅读量。默认每页 10 条，支持分页导航 |
| FR-L02 | P0 | **分页**：列表底部提供上一页/下一页及页码跳转，URL 体现分页参数（如 `/page/2`），支持 SEO 友好的分页链接 |
| FR-L03 | P1 | **置顶文章**：博主标记为"置顶"的文章始终排在列表最前，并带有置顶标识 |
| FR-L04 | P0 | **左侧分类列表**：首页左侧栏展示所有分类，每个分类显示名称及该分类下的文章数量，点击某一分类后右侧文章列表仅显示该分类下的文章，URL 体现筛选条件（如 `/category/技术`） |
| FR-L05 | P0 | **左侧标签列表**：首页左侧栏展示所有标签，每个标签显示名称及该标签下的文章数量，点击某一标签后右侧文章列表仅显示该标签下的文章，URL 体现筛选条件（如 `/tag/Go`） |
| FR-L06 | P2 | **列表布局切换**：支持在"标准列表"和"卡片网格"两种布局间切换，用户偏好通过 localStorage 记忆，全局生效（首页及分类/标签/搜索筛选页共享同一偏好） |
| FR-L07 | P2 | **滚动加载**：除传统分页外，支持向下滚动自动加载更多（无限滚动模式），可作为用户可切换选项 |

#### 2.1.2 文章详情页

文章详情页是核心阅读页面，承载正文展示与阅读辅助功能。

| 编号 | 优先级 | 需求描述 |
|------|--------|----------|
| FR-D01 | P0 | **正文渲染**：将 Markdown 内容渲染为 HTML 展示，支持标题层级（H1-H6）、加粗、斜体、删除线、引用块、有序/无序列表、表格、分割线、行内代码与代码块、图片、超链接等标准 Markdown 语法 |
| FR-D02 | P0 | **代码高亮**：代码块按语言进行语法高亮，支持常见编程语言（Go、Python、JavaScript、TypeScript、Java、Rust、Shell、SQL、YAML、Dockerfile、HTML、CSS、SCSS、JSON、XML 等），并提供行号显示与一键复制按钮 |
| FR-D03 | P0 | **文章元信息**：页面头部显示文章标题、作者名、发布时间（格式：YYYY-MM-DD）、最后更新时间、所属分类、标签列表，标签可点击跳转至对应标签筛选页 |
| FR-D04 | P0 | **目录导航（TOC）**：根据 Markdown 标题自动生成文章目录（Table of Contents），以侧栏或浮层形式展示，支持点击跳转到对应章节，当前阅读位置高亮 |
| FR-D05 | P1 | **双向链接**：若文章正文引用了其他站内文章，自动生成引用链接；被引用文章页面底部展示"被以下文章引用"的反向链接列表。外部链接自动添加 `rel="noopener noreferrer"` 安全属性 |
| FR-D06 | P1 | **上一篇/下一篇**：文章底部提供上一篇和下一篇导航，按发布时间顺序排列，方便连续阅读 |
| FR-D07 | P1 | **版权声明**：文章底部展示博主预设的版权声明（如 CC 协议），可全局配置或单篇覆盖 |
| FR-D08 | P1 | **社交分享**：提供分享按钮（复制链接、Twitter/X 等），生成 Open Graph meta 标签以优化社交平台预览。注意：Open Graph 标签生成由 NFR-SE03（P1）约束，两者在 v1.1 中同步实现 |
| FR-D09 | P2 | **预计阅读时间**：根据正文字数自动计算预计阅读时间（如"约 5 分钟"），展示在文章头部 |
| FR-D10 | P2 | **打印样式**：提供适合打印的 CSS 样式，隐藏导航栏、侧栏等无关元素 |

### 2.2 后台管理（博主端）

后台是博主的内容管理中心，所有操作需登录认证后方可使用。

#### 2.2.1 文章管理

文章管理是后台核心模块，覆盖文章从创建到发布的完整生命周期。

| 编号 | 优先级 | 需求描述 |
|------|--------|----------|
| FR-AM01 | P0 | **文章列表**：后台文章列表展示所有文章（含草稿、私密发布、公开发布），每行显示标题、状态（草稿/私密/公开）、分类、标签、创建时间、更新时间，支持分页 |
| FR-AM02 | P0 | **创建文章**：提供 Markdown 编辑器，支持填写标题、正文、分类（单选）、标签（多选），可保存为草稿或直接发布 |
| FR-AM03 | P0 | **编辑文章**：打开已有文章进入编辑器，支持修改所有字段，修改后支持更新发布或存为草稿 |

## 3. 界面设计

> 本章定义 zblog 的视觉风格与页面交互规范。设计原则遵循：**极简克制**（降低认知负荷）、**内容优先**（以阅读体验为核心）、**开发者审美**（符合技术用户群体的审美偏好）。

---

### 3.1 设计风格

#### 3.1.1 整体定位

zblog 面向技术开发人员，设计风格定位为 **极简主义技术风**，关键词：克制、留白、清晰、聚焦。

参考风格：
- [Bear Blog](https://bearblog.dev/) 的极简排版
- [GitHub](https://github.com/) 的界面克制感
- [Medium](https://medium.com/) 的阅读舒适度

#### 3.1.2 色彩方案

| 用途 | 颜色值 | 说明 |
|------|--------|------|
| 主色调 | `#1a1a2e` 深蓝黑 | 正文、标题、强调元素 |
| 辅色调 | `#e94560` 珊瑚红 | 链接、按钮、高亮、标签强调。⚠ 在白色背景上对比度约 4.3:1，略低于 WCAG AA 4.5:1 标准；正文内链接可考虑降深为 `#d1334a` 以满足对比度要求（对应 NFR-A10） |
| 背景色 | `#fafafa` 极浅灰 | 页面整体背景 |
| 卡片背景 | `#ffffff` 纯白 | 文章卡片、内容区块 |
| 边框色 | `#e1e4e8` 浅灰 | 分割线、卡片边框 |
| 次要文字 | `#6a737d` 中灰 | 日期、摘要、辅助信息 |
| 代码块背景 | `#f6f8fa` 浅蓝灰 | 行内代码与代码块底色 |

> 后续可考虑支持暗色模式（P3），暗色主题色彩另行定义。

#### 3.1.3 字体方案

| 用途 | 字体栈 | 说明 |
|------|--------|------|
| 正文字体 | `-apple-system, BlinkMacSystemFont, "Segoe UI", "Noto Sans SC", "PingFang SC", "Microsoft YaHei", sans-serif` | 系统默认无衬线字体，优先中文可读性 |
| 代码字体 | `"JetBrains Mono", "Fira Code", "Cascadia Code", "Source Code Pro", Consolas, monospace` | 等宽字体，优先支持连字（ligatures）的编程字体 |
| 标题字体 | 同正文字体，字重加粗 | 保持统一，不过度装饰 |

字体大小阶梯（Type Scale）：

| 层级 | 大小 | 用途 |
|------|:---:|------|
| H1 | 2rem（32px） | 文章详情页标题 |
| H2 | 1.5rem（24px） | 文章内一级标题 |
| H3 | 1.25rem（20px） | 文章内二级标题 |
| H4 | 1.1rem（18px） | 文章内三级标题 |
| 正文 | 1rem（16px） | 正文字号 |
| 小字 | 0.875rem（14px） | 日期、摘要、辅助信息 |
| 代码 | 0.9rem（14.4px） | 代码块字号 |

行高：正文 1.75，标题 1.4，代码块 1.5。

#### 3.1.4 间距系统

基于 4px 基准的间距体系：

| Token | 值 | 用途 |
|------|:---:|------|
| xs | 4px | 紧密元素间距 |
| sm | 8px | 相关元素间距 |
| md | 16px | 段落间距、卡片内边距 |
| lg | 24px | 区块间距 |
| xl | 32px | 大区块间距 |
| 2xl | 48px | 页面级分隔 |
| 3xl | 64px | 页面首尾留白 |

内容区最大宽度：主内容区 **960px**（左侧栏 220px + 右侧文章列表 700px，间距 40px）。

### 3.2 页面结构

#### 3.2.1 全局布局

所有前台页面采用统一布局骨架：

```
┌───────────────────────────────────────────────────┐
│  导航栏（固定顶部）                                  │
│  Logo    首页  归档  分类  标签  关于  🔍            │
├───────────────────────────────────────────────────┤
│                  主内容区（max-width: 960px）        │
│                                                   │
├───────────────────────────────────────────────────┤
│  页脚                                              │
│  © 2026 · Powered by zblog · RSS · 备案号           │
└───────────────────────────────────────────────────┘
```

- 导航栏固定在页面顶部（sticky），背景半透明 + 毛玻璃效果
- 主内容区水平居中，最大宽度 960px
- 移动端导航折叠为汉堡菜单

#### 3.2.2 首页（文章列表页）

首页采用左右双栏布局：左侧为固定导航侧栏（分类 + 标签），右侧为文章卡片列表。

```
┌───────────────────────────────────────────────────┐

## 4. 非功能需求

> 非功能需求定义了系统的质量属性与约束条件，贯穿所有功能模块。以下从性能、安全、可用性、可维护性、SEO 五个维度展开，每个维度以 **NFR 编号**（Non-Functional Requirement）标识，优先级与功能需求保持一致。

---

### 4.1 性能要求

系统性能直接影响用户体验与搜索引擎排名，以下指标基于目标用户画像（技术开发者，通常网络环境良好）制定。

#### 4.1.1 响应时间

| 编号 | 优先级 | 需求描述 |
|------|--------|----------|
| NFR-P01 | P0 | **首屏加载时间**：前台页面（首页、详情页、归档页）从发起请求到首屏内容渲染完成 ≤ 1.5 秒（LCP ≤ 2.5s，基于 Web Vitals 标准），测试条件为 4G 网络环境 |
| NFR-P02 | P0 | **页面切换**：前台页面间导航切换的完整加载时间 ≤ 1 秒，测试条件为同源页面间导航（非跨域）、4G 网络环境、浏览器缓存已预热（即首次访问后再次导航）。利用浏览器缓存（304 Not Modified）和预加载（`<link rel="prefetch">`）优化常用页面间的导航体验 |
| NFR-P03 | P1 | **搜索响应**：全文搜索请求响应时间 ≤ 500ms（1000 篇文章以内数据量） |
| NFR-P04 | P1 | **后台操作**：文章保存、发布、删除等后台操作响应时间 ≤ 1 秒 |
| NFR-P05 | P1 | **图片加载**：文章内图片采用懒加载（Lazy Loading），首屏仅加载可视区域图片 |
| NFR-P06 | P2 | **TTI（Time to Interactive）**：页面可交互时间 ≤ 3 秒 |

#### 4.1.2 并发与吞吐量

| 编号 | 优先级 | 需求描述 |
|------|--------|----------|
| NFR-P07 | P1 | **并发支撑**：系统应能支撑至少 50 个并发请求无性能劣化（单机部署场景） |
| NFR-P08 | P1 | **静态资源**：CSS/JS/图片等静态资源支持 Gzip/Brotli 压缩，Cache-Control 缓存头设置合理的过期时间 |
| NFR-P09 | P2 | **峰值应对**：支持突发流量（如文章被 HN/Reddit 推荐）时的基本可用性，可通过 CDN 缓存静态页面 |

#### 4.1.3 缓存策略

| 编号 | 优先级 | 需求描述 |
|------|--------|----------|
| NFR-P10 | P0 | **页面缓存**：公开文章详情页支持服务端页面缓存（如内存缓存或文件缓存），缓存整个渲染后的 HTML 页面（含模板布局），缓存失效时间为文章更新时主动清除。注意：与 `posts.content_html`（仅正文预渲染）不同，页面缓存包含完整 HTML 输出 |
| NFR-P11 | P1 | **查询缓存**：分类列表、标签列表、归档页等低频变更数据支持缓存，缓存时长可配置（默认 10 分钟） |
| NFR-P12 | P2 | **CDN 加速**：静态资源（CSS、JS、字体、图片）支持通过 CDN 分发 |

### 4.2 安全性要求

zblog 面向个人部署使用，安全需求以防范常见 Web 攻击和保护博主数据为核心。

#### 4.2.1 Web 安全防护

| 编号 | 优先级 | 需求描述 |
|------|--------|----------|
| NFR-S01 | P0 | **XSS 防护**：所有用户输入（评论内容、搜索关键词等）在输出渲染前必须进行 HTML 实体编码转义；文章正文 Markdown 渲染后需经过 HTML 清洗（sanitize），移除危险标签（`<script>`、`<iframe>`、`<object>` 等）和事件属性（`onerror`、`onload` 等） |
| NFR-S02 | P0 | **CSRF 防护**：后台所有状态变更操作（发布/编辑/删除文章、修改设置等）需验证 CSRF Token，Token 绑定用户会话 |
| NFR-S03 | P0 | **SQL 注入防护**：所有数据库操作必须使用参数化查询或 ORM 提供的安全查询方法，禁止拼接 SQL 字符串 |
| NFR-S04 | P0 | **密码安全**：博主密码使用 bcrypt / argon2 等自适应哈希算法加密存储，禁止明文或弱哈希（MD5/SHA1）存储 |
| NFR-S05 | P0 | **HTTPS**：部署环境必须启用 HTTPS/TLS 1.2+，HTTP 请求强制 301 重定向至 HTTPS；设置 HSTS 头 |
| NFR-S06 | P1 | **内容安全策略（CSP）**：设置 Content-Security-Policy 头，限制脚本来源、样式来源、图片来源等，降低 XSS 影响范围 |
| NFR-S07 | P1 | **文件上传安全**：上传文件需校验 MIME 类型与文件扩展名一致性，禁止上传可执行文件（.exe、.php、.sh 等）；上传目录禁止脚本执行权限 |
| NFR-S08 | P1 | **敏感信息保护**：错误页面不暴露堆栈信息、数据库结构、框架版本等敏感细节；生产环境关闭 Debug 模式 |

#### 4.2.2 认证与授权

| 编号 | 优先级 | 需求描述 |
|------|--------|----------|
| NFR-S09 | P0 | **会话安全**：登录会话使用 HttpOnly + Secure + SameSite=Lax 属性的 Cookie 承载，防止 JavaScript 读取和 CSRF 利用 |
| NFR-S10 | P1 | **登录保护**：连续 5 次登录失败后，对该 IP/账户实施 15 分钟锁定或要求输入验证码 |
| NFR-S11 | P1 | **密码复杂度**：密码长度 ≥ 8 位，需包含字母和数字（在修改密码时校验）。注：v1.0 安装引导（FR-DM04 P0）中已内置同等密码强度校验，NFR-S11 作为 P1 需求扩展至后台密码修改场景，确保全生命周期密码策略一致 |
| NFR-S12 | P2 | **二因子认证**：支持 TOTP 二因子认证，绑定后登录需输入 6 位动态验证码（对应 FR-UA06） |

#### 4.2.3 依赖安全

| 编号 | 优先级 | 需求描述 |
|------|--------|----------|
| NFR-S13 | P1 | **依赖更新**：使用的第三方库/框架应保持更新，及时修复已知安全漏洞（CVE），建议使用依赖扫描工具（后端：Dependabot / OWASP Dependency-Check Gradle/Maven 插件；前端：Dependabot / npm audit） |
| NFR-S14 | P2 | **最小权限原则**：数据库连接使用最低必要权限账户（仅访问本应用数据库，无需 DROP/CREATE DATABASE 权限）；文件系统仅开放必要目录的读写权限 |

## 5. 技术方案

> 本章给出 zblog 的推荐技术选型、系统架构、数据库设计和 API 规范。技术选型以 **简洁、稳定、易维护** 为首要原则，优先选择社区成熟、文档丰富的技术栈，降低个人长期维护成本。

### 5.1 技术选型

#### 5.1.1 选型原则

| 原则 | 说明 |
|------|------|
| **简单优先** | 避免过度工程化，能用内置功能解决的问题不引入额外依赖 |
| **标准优先** | 优先使用语言/框架标准库或社区共识方案，降低学习曲线 |
| **维护优先** | 选择长期维护、社区活跃的项目，避免小众或无维护的依赖 |
| **文件型优先** | 默认使用 SQLite 零运维存储，同时支持 MySQL/PostgreSQL 以满足不同部署场景 |
| **前后端分离** | 后端提供 RESTful API，前端 React 构建独立的前台展示与后台管理 SPA |

#### 5.1.2 推荐技术栈

#### 5.1.3 架构决策说明

**为何选择前后端分离 + React？**

zblog 的交互复杂度集中在后台管理端（Markdown 编辑器、媒体库、分类标签管理、站点设置等），这些场景需要丰富的客户端交互，传统的服务端模板渲染方式实现成本高、用户体验差。React 能为后台提供流畅的 SPA 体验，而前台展示仍可选用服务端渲染以保证 SEO。

**为何同时支持三种数据库？**

- **SQLite**：面向"零运维"场景，下载即可用，无需安装数据库服务，适合个人 VPS 极简部署
- **MySQL**：面向已有 MySQL 环境的用户（如已购买 cPanel 虚拟主机），直接复用现有数据库服务
- **PostgreSQL**：面向追求功能丰富性的用户，后续版本可能利用 PostgreSQL JSONB、全文搜索等高级特性

Spring Data JPA + Hibernate 屏蔽了底层数据库差异，通过 Flyway 管理各数据库的建表脚本，用户只需在配置文件中切换 `spring.datasource` 即可。

**前台渲染方案的选择：**

| 方案 | 优点 | 缺点 | 推荐场景 |
|------|------|------|---------|
| **Thymeleaf（推荐）** | SEO 友好、首屏性能最优、无需 Node.js 运行时 | 前台无法复用 React 组件 | 追求极简、SEO 优先 |
| **Next.js SSR** | 前后端统一技术栈、React 生态丰富 | 需要 Node.js 运行时、增加运维复杂度 | 追求前后端统一 |

v1.0 推荐使用 **Thymeleaf** 渲染前台页面（SEO 优先），后台管理使用 React SPA。

#### 5.1.4 不推荐引入的技术

| 技术 | 不推荐理由 |
|------|-----------|
| Spring Cloud / 微服务 | 个人博客不需要分布式架构，单体 Spring Boot 已足够 |
| Redis / Memcached | 个人博客数据量下，JVM 本地缓存（Caffeine）或文件缓存已足够 |
| GraphQL | RESTful API 更简单直接，GraphQL 增加前后端复杂度 |
| Redux / MobX | 博客后台状态管理简单，React Context + useReducer 或 Zustand 足够 |
| Kotlin（非备选方案） | 除非团队有 Kotlin 经验，否则 Java 生态更广、文档更丰富 |
| Webpack | Vite 开发体验更好，已成为 React 社区主流选择 |
| Angular / Vue | 增加学习成本和社区分裂，统一使用 React 减少技术栈宽度 |

### 5.2 系统架构

#### 5.2.1 架构概览

zblog 采用 **前后端分离的单体架构**，后端 Spring Boot 提供 RESTful API + 前台页面渲染（Thymeleaf），前端 React 构建后台管理 SPA：

│  │  (Thymeleaf 服务端渲染)  │  │  - Markdown 编辑器           │ │
│  │  - SEO 友好             │  │  - 媒体库管理                │ │
│  │  - 首屏性能优            │  │  - 站点设置                  │ │
│  └───────────┬────────────┘  └──────────────┬──────────────┘ │

#### 5.2.2 前后端交互模式

zblog 采用 **双模式交互**，前台与后台使用不同的渲染策略：

| 端 | 渲染方式 | 数据交互 | 路由 |
|----|---------|---------|------|
| **前台（读者端）** | Thymeleaf 服务端模板渲染 | 服务端直接渲染 HTML 返回 | 服务端路由（`/`、`/post/{slug}` 等） |
| **后台管理（博主端）** | React SPA 客户端渲染 | 通过 RESTful API（JSON）交互 | 前端路由（React Router），入口 `/admin` |

```
前台请求流程：
  浏览器 → GET /post/my-article
         → Spring MVC Controller
         → Service 层获取数据
         → Thymeleaf 模板渲染 HTML
         → 返回完整 HTML 页面（SEO 友好）

后台请求流程：
  浏览器 → 加载 /admin（React SPA 入口）
         → React Router 接管路由
         → 组件内 fetch GET /api/v1/admin/posts
         → Spring MVC Controller（返回 JSON）
         → React 组件渲染 UI
```

### 5.3 数据库设计

#### 5.3.1 设计原则

- **多数据库兼容**：默认使用 SQLite，同时兼容 MySQL 8.0+ 和 PostgreSQL 15+。通过 Flyway 为每种数据库提供独立的迁移脚本
- **JPA 实体优先**：使用 Spring Data JPA 注解定义实体，由 Hibernate 自动建表（dev 环境 `ddl-auto: update`），生产环境使用 Flyway 迁移脚本管理
- **主键策略**：使用自增整数 ID。SQLite 用 `INTEGER PRIMARY KEY AUTOINCREMENT`；MySQL 用 `BIGINT AUTO_INCREMENT`；PostgreSQL 用 `BIGSERIAL`
- **时间字段**：Java 端使用 `LocalDateTime` 类型，数据库统一存储 ISO 8601 格式（TEXT 类型），确保跨数据库一致
- **软删除**：使用 `deleted_at` 字段（NULL = 未删除），而非物理删除
- **外键约束**：JPA `@ManyToOne` / `@OneToMany` 定义关联关系。SQLite 需 `PRAGMA foreign_keys = ON`（Spring Boot 启动时自动设置）
- **数据库切换**：通过 Spring Profile（`sqlite` / `mysql` / `pg`）切换，Flyway 自动选择对应迁移脚本

| 字段 | 类型 | 约束 | 说明 |
|------|------|------|------|

| slug | TEXT | NOT NULL, UNIQUE | URL 标识，自动生成或手动指定 |
| content | TEXT | NOT NULL | Markdown 原始内容 |
| content_html | TEXT | NOT NULL | 渲染后的 HTML 内容（保存时预渲染并持久化存储，区别于 NFR-P10 的运行时页面缓存） |

### 5.4 API 设计

#### 5.4.1 设计规范

| 规范项 | 约定 |
|--------|------|
| **风格** | RESTful，资源导向 URL |
| **数据格式** | JSON（请求/响应），Content-Type: `application/json` |
| **认证方式** | Cookie + Session（后台 API）；前台公开 API 无需认证 |
| **版本控制** | URL 前缀 `/api/v1/`，大版本变更时递增 |
| **命名规范** | URL 使用 kebab-case，JSON 字段统一使用 snake_case（如 `created_at`、`content_html`、`page_size`、`is_pinned`），与数据库字段命名保持一致，通过 Jackson 全局配置 `PropertyNamingStrategies.SNAKE_CASE` 实现，避免混用 camelCase 和 snake_case 导致的命名不一致问题 |
| **分页规范** | 请求参数 `?page=1&page_size=10`，响应含 `total`、`page`、`page_size`、`total_pages` |
| **错误响应** | 统一格式 `{ "error": { "code": "...", "message": "..." } }` |
| **HTTP 状态码** | 200（成功）、201（创建成功）、204（无内容，删除成功）、400（参数错误）、401（未认证）、403（无权限）、404（资源不存在）、410（资源已删除）、422（业务逻辑错误）、500（服务端错误） |

#### 6.1.2 Web 安装引导（FR-DM04）

zblog 提供 Web 图形化安装引导流程，用户启动 JAR 后通过浏览器访问即可完成数据库初始化，无需手动编辑配置文件或执行命令行迁移脚本。

**触发条件：**

系统启动时自动检测数据库连接状态，满足以下任一条件即进入安装引导模式：

- 未配置数据源（`application.yml` 中缺少数据库连接信息）
- 已配置数据库但连接失败（连接超时、认证失败、数据库不存在等）
- 数据库已连接但表结构未初始化（首次启动）

**安装引导流程：**

```

  │
  ▼

**安全设计：**

| 措施 | 说明 |
|------|------|
| 安装锁定 | 安装完成后写入锁定标识，后续访问 `/install` 返回 404，防止重复初始化 |
| 密码强度 | 管理员密码要求 ≥ 8 位，需包含字母和数字（与 NFR-S11 P1 一致，v1.0 安装引导中即强制执行，确保初始密码安全基线。注意：NFR-S11 作为 P1 需求定义的是全局密码复杂度策略——含修改密码时的校验；安装引导中的密码校验是 FR-DM04 P0 的内置安全要求，两者独立但标准统一） |
| CSRF 保护 | 安装表单携带 CSRF Token。由于安装引导阶段尚无用户登录 Session，采用 **Double Submit Cookie** 模式：服务端生成随机 Token，同时写入 Cookie 和表单隐藏字段，提交时比对两者一致性。Token 在安装引导 Session（临时会话，非登录 Session）范围内有效 |
| 文件权限 | 安装后建议将配置文件设为只读（`chmod 400 application.yml`） |
| 敏感信息 | 数据库密码写入配置后，安装引导页不可再次访问 |

**安装引导页面实现说明**：安装引导页面的 Controller（`InstallController`）不依赖任何 JPA Repository 或 Service 层组件，仅返回独立的 Thymeleaf 模板。`InstallDetectFilter`（数据库检测拦截器）需将 `/install` 路径及其静态资源（CSS/JS）明确列入放行白名单，确保安装引导页面在任何数据库状态下均可正常访问。安装引导使用的 Session 为临时会话，与后台登录 Session 隔离，安装完成后临时 Session 自动失效。

**安装引导页面样式：**

安装引导页面使用独立的简洁模板，不依赖数据库，纯静态 Thymeleaf 渲染，包含：
- 步骤进度条（步骤 1/3 → 2/3 → 3/3）
- 表单校验（前端 JS + 后端双重校验）
- 错误提示（连接失败、迁移失败等友好报错信息）
- 响应式布局（支持移动端完成安装）

| 网络 | 公网 IP + 开放 80/443 端口 | — |

**部署步骤（概述）：**

1. **安装 Java**：`apt install openjdk-17-jdk-headless`
2. **获取 JAR**：从 GitHub Releases 下载 `zblog-1.0.0.jar`
3. **创建目录结构**：
   ```
   /opt/zblog/
   ├── zblog.jar        # Spring Boot Fat JAR
   ├── application.yml  # 外部配置文件（可选，覆盖 JAR 内配置）
   ├── data/            # 数据目录
   │   ├── zblog.db     # SQLite 数据库文件（默认）
   │   ├── uploads/     # 上传文件目录
   │   ├── cache/       # 缓存文件目录
   │   └── logs/        # 日志文件目录
   └── static/          # 静态资源（可选，外部覆盖）
   ```
4. **配置数据库**（`application.yml` 或环境变量）：
   ```yaml
   # SQLite（默认，无需额外配置）
   spring:
     profiles:
       active: sqlite
     datasource:
       url: jdbc:sqlite:data/zblog.db
       driver-class-name: org.sqlite.JDBC

   # 或切换到 MySQL
   # spring:
   #   profiles:
   #     active: mysql
   #   datasource:
   #     url: jdbc:mysql://localhost:3306/zblog
   #     username: zblog
   #     password: ${DB_PASSWORD}

   # 或切换到 PostgreSQL
   # spring:
   #   profiles:
   #     active: pg
   #   datasource:
   #     url: jdbc:postgresql://localhost:5432/zblog
   #     username: zblog
   #     password: ${DB_PASSWORD}
   ```
5. **创建 systemd 服务**：
   ```ini
   # /etc/systemd/system/zblog.service
   [Unit]
   Description=zblog personal blog
   After=network.target

   [Service]
   Type=simple
   User=zblog
   WorkingDirectory=/opt/zblog
   ExecStart=/usr/bin/java -Xmx256m -jar /opt/zblog/zblog.jar
   Restart=on-failure
   RestartSec=10

   [Install]
   WantedBy=multi-user.target
   ```
6. **启动服务**：`systemctl enable --now zblog`
7. **配置反向代理**（Nginx 或 Caddy）处理 HTTPS，代理到 `127.0.0.1:8080`

### 6.4 备份与恢复

#### 6.4.1 备份策略

zblog 数据由两部分组成，备份策略因数据库类型而异：

| 备份对象 | SQLite | MySQL | PostgreSQL | 频率 |
|------|--------|-------|------------|------|
| **数据库** | 复制 `data/zblog.db`（`.backup` 或 `cp`） | `mysqldump` 导出 SQL | `pg_dump` 导出 SQL | 每日 |
| **上传文件** | `data/uploads/` 目录 | `data/uploads/` 目录 | `data/uploads/` 目录 | 每日 |
| **配置文件** | `application.yml`、systemd service | 同左 | 同左 | 变更时 |

**SQLite 在线备份命令：**

```bash
# 方法1：SQLite .backup 命令（安全，不阻塞读写）
sqlite3 /opt/zblog/data/zblog.db ".backup '/backup/zblog_$(date +%Y%m%d).db'"

# 方法2：文件复制（需确保无写入事务）
cp /opt/zblog/data/zblog.db /backup/zblog_$(date +%Y%m%d).db
```

**MySQL 备份命令：**
```bash
mysqldump -u zblog -p zblog > /backup/zblog_$(date +%Y%m%d).sql
```

**PostgreSQL 备份命令：**
```bash
pg_dump -U zblog zblog > /backup/zblog_$(date +%Y%m%d).sql
```

#### 6.4.2 备份脚本示例结构（SQLite 模式）

```bash
#!/bin/bash
# /opt/zblog/scripts/backup.sh
BACKUP_DIR="/backup/zblog"
DATE=$(date +%Y%m%d_%H%M%S)
mkdir -p "$BACKUP_DIR"

# 备份数据库
sqlite3 /opt/zblog/data/zblog.db ".backup '$BACKUP_DIR/zblog_$DATE.db'"

# 备份上传文件
tar -czf "$BACKUP_DIR/uploads_$DATE.tar.gz" -C /opt/zblog/data uploads/

# 清理 30 天前的旧备份
find "$BACKUP_DIR" -name "*.db" -mtime +30 -delete
find "$BACKUP_DIR" -name "*.tar.gz" -mtime +30 -delete
```

配置 crontab 每日凌晨执行：
```
0 3 * * * /opt/zblog/scripts/backup.sh >> /var/log/zblog_backup.log 2>&1
```

#### 6.4.3 异地备份

| 方案 | 说明 | 成本 |
|------|------|:---:|
| **rsync 到另一台服务器** | 最简单，适合有备用 VPS 的用户 | 低 |
| **云对象存储** | AWS S3 / 阿里云 OSS / 腾讯云 COS / Cloudflare R2，使用 rclone 或 s3cmd 同步 | 低 |
| **GitHub 私有仓库** | 适用于仅备份数据库文件（几十 MB 以内） | 免费 |

#### 6.4.4 恢复流程

**SQLite 模式：**
1. **停止服务**：`systemctl stop zblog`
2. **恢复数据库**：将备份的 `.db` 文件覆盖到 `data/zblog.db`
3. **恢复上传文件**：解压上传文件备份到 `data/uploads/`
4. **启动服务**：`systemctl start zblog`
5. **验证**：访问首页和后台，确认数据完整

**MySQL / PostgreSQL 模式：**
1. **停止服务**：`systemctl stop zblog`
2. **恢复数据库**：`mysql -u zblog -p zblog < backup.sql` 或 `psql -U zblog zblog < backup.sql`
3. **恢复上传文件**：同 SQLite 模式
4. **启动服务**：`systemctl start zblog`

#### 6.4.5 灾难恢复方案

| 场景 | 恢复方案 | RTO（恢复时间目标） | RPO（恢复点目标） |
|------|---------|:---:|:---:|
| 误删文章 | 从文章版本历史恢复（FR-AM12） | 5 分钟 | 无丢失 |
| 数据库损坏 | 从最新备份恢复 | 30 分钟 | 最近 24 小时 |
| 服务器宕机 | 在新服务器重新部署 + 恢复备份 | 1 小时 | 最近 24 小时 |
| 域名/备案失效 | 更换域名，更新配置中的站点 URL | 2 小时 | 无数据丢失 |

## 8. 附录

### 8.1 竞品参考

以下为当前主流的个人博客/知识管理方案，zblog 在设计和选型过程中参考了这些项目的优点，同时针对其不足进行了差异化设计。

#### 8.1.1 动态博客系统

| 项目 | 技术栈 | 优点 | 不足 | zblog 差异化 |
|------|--------|------|------|-------------|
| **WordPress** | PHP + MySQL | 生态最丰富、插件海量、主题多、社区庞大 | 臃肿、性能开销大、需维护 PHP+MySQL、安全漏洞多、不适合开发者 | SQLite 默认零运维/MySQL/PG 多数据库可选、Markdown 原生支持、Java 企业级稳定 |
| **Ghost** | Node.js + MySQL | 专业博客体验、编辑器优秀、SEO 好、会员系统 | 依赖 Node.js 生态、资源占用高、升级复杂 | Spring Boot 单体应用、多数据库兼容、React 现代化后台 |
| **Halo** | Java + MySQL | 国内生态好、主题多、界面现代 | 仅支持 MySQL、部署较重、面向多用户而非个人知识库 | SQLite 默认零运维、多数据库可选、React 后台 + Thymeleaf 前台分离、面向个人知识管理 |

#### 8.1.2 静态站点生成器（SSG）

| 项目 | 技术栈 | 优点 | 不足 | zblog 差异化 |
|------|--------|------|------|-------------|
| **Hugo** | Go | 构建速度极快、单二进制、社区活跃 | 需要重新构建才能更新内容、无在线编辑器、非技术人员难上手 | 在线编辑器、即时发布、无需构建流程、多数据库可选 |
| **Hexo** | Node.js | 生态丰富、主题多、中文社区活跃 | 构建速度随内容增长变慢、Node.js 依赖 | Java 企业级性能、在线管理、动态搜索 |
| **Astro** | JavaScript | 现代架构、支持多框架组件、性能优秀 | 仍是 SSG 模式、需要构建/部署流程 | 动态渲染、实时更新、Spring Boot 内嵌服务器 |

#### 8.1.3 笔记/知识管理工具

| 项目 | 技术栈 | 优点 | 不足 | zblog 差异化 |
|------|--------|------|------|-------------|
| **Notion** | 云端 SaaS | 功能强大、数据库视图、协作 | 非自托管、数据不自主、加载慢、闭源 | 数据完全自主可控、开放源码 |
| **Obsidian** | Electron | 本地优先、双向链接强、插件生态 | 非 Web 原生、需客户端、分享不便 | Web 原生、天然可分享、公开/私密灵活 |
| **Logseq** | ClojureScript | 开源、大纲式笔记、双向链接、本地文件 | 非传统博客形态、分享能力弱 | 博客形态、兼顾写作与分享 |
| **SiYuan（思源）** | Go + Electron | 国产开源、本地优先、块级引用 | 非 Web 博客形态、对外分享弱 | 专注博客/知识库对外展示场景 |

## 9. 风险评估

> 本章识别 zblog 项目在技术、进度、需求层面面临的关键风险，并给出应对措施。风险等级 = 发生概率 × 影响程度，分为 **高（H）**、**中（M）**、**低（L）** 三级。

### 9.1 风险清单

| 编号 | 风险描述 | 类别 | 概率 | 影响 | 等级 | 应对措施 |
|------|---------|------|:---:|:---:|:---:|---------|
| RISK-01 | **多数据库兼容增加复杂度**：SQLite/MySQL/PostgreSQL 三种数据库需分别维护 Flyway 迁移脚本和 SQL Dialect 适配，开发与测试工作量翻倍 | 技术 | 高 | 中 | **H** | (1) v1.0 优先保证 SQLite 完整性，MySQL/PG 作为"尽力兼容"，测试覆盖核心 CRUD 场景；(2) 利用 Spring Data JPA + Hibernate Dialect 屏蔽大部分差异；(3) CI 中加入多数据库矩阵测试 |
| RISK-02 | **评论系统 v1.1 延期风险**：v1.1 需同时交付评论前台展示、后台审核、回复、反垃圾、评论者信息等多个子功能（6 项 P1），工作量集中 | 进度 | 中 | 中 | **M** | (1) 评论模块拆分为两个子迭代：v1.1a（前台提交+后台审核+回复+评论者表单）→ v1.1b（反垃圾过滤）；(2) 基础反垃圾仅实现关键词黑名单+频率限制，不引入外部服务 |
| RISK-03 | **单人开发资源瓶颈**：项目估算基于单人业余开发（每周 15-20h），如遇突发事务或技术难题，MVP 交付可能延期 | 进度 | 高 | 高 | **H** | (1) 严格遵循 MVP 范围，拒绝需求蔓延（scope creep）；(2) 每周进度回顾，如偏差 > 20% 则触发应对计划（削减 P1 或延长工期）；(3) 优先使用成熟方案，减少自研轮子 |
| RISK-04 | **前台 Thymeleaf + 后台 React 双技术栈**：前台使用服务端模板渲染，后台使用 React SPA，两套渲染体系增加维护成本 | 技术 | 低 | 低 | **L** | 可接受风险。两套体系各司其职（SEO vs 复杂交互），边界清晰；v1.0 不混用，不存在耦合问题。远期可视情况将前台迁移至 Next.js SSR |
| RISK-05 | **Web 安装引导安全风险**：`/install` 路由在未安装状态下对外开放，可能被恶意访问者抢先完成安装并设置管理员密码 | 安全 | 中 | 高 | **H** | (1) 安装引导仅在数据库未初始化时启用，安装完成后立即锁定（写入 `.zblog_installed` 标记文件）；(2) 首次访问 `/install` 时生成一次性安装 Token（存储在内存中，重启失效）；(3) 部署文档中强调：在公网环境部署时应先通过防火墙限制 `/install` 路径访问，或使用反向代理 IP 白名单 |
| RISK-06 | **搜索性能退化**：v1.0 使用数据库内置全文搜索（SQLite FTS5 / MySQL FULLTEXT / PG tsvector），文章量增长到数千篇后搜索响应可能超过 500ms 目标（NFR-P03） | 技术 | 低 | 中 | **L** | (1) v1.0 目标数据量为个人博客规模（< 1000 篇），数据库内置搜索足够；(2) v1.2 评估是否引入 Lucene/Hibernate Search 独立索引；(3) 搜索结果分页 + 缓存降低单次查询开销 |
| RISK-07 | **跨浏览器兼容性测试不足**：NFR-A01 承诺支持 Chrome/Firefox/Safari/Edge 最新两版，但单人开发难以覆盖全部浏览器和设备的测试 | 质量 | 高 | 中 | **H** | (1) 开发阶段使用 Chrome 为主浏览器，Edge（Chromium 内核）天然兼容；(2) M2 测试阶段使用 BrowserStack 或 Playwright 进行跨浏览器回归测试；(3) 优先保证核心阅读流程（首页→详情页→搜索）在三款浏览器中可用 |

### 9.3 高风险项专项应对计划

#### RISK-01：多数据库兼容复杂度

- **触发条件**：Flyway 迁移脚本在三种数据库上行为不一致，或 SQL Dialect 适配出现未预期的兼容性问题
- **应对负责人**：后端开发
- **应对时间窗口**：M1 开发阶段 Week 1-2（数据库基础设施搭建期）
- **应急方案**：若 MySQL/PG 兼容工作量超过 3 人天，v1.0 仅交付 SQLite 模式，MySQL/PG 延至 v1.1

#### RISK-03：单人开发资源瓶颈

- **触发条件**：连续 2 周实际投入 < 10 小时/周，或进度偏差 > 30%
- **应对负责人**：项目经理（即开发者本人）
- **应对时间窗口**：整个 M1 开发阶段
- **应急方案**：(1) 砍 P1 功能，聚焦 P0 核心闭环；(2) 延长 M1 工期至 8 周（原计划 4-6 周上限）；(3) 寻求开源社区贡献者协助

#### RISK-05：Web 安装引导安全

- **触发条件**：部署至公网环境，且未配置反向代理 IP 白名单
- **应对负责人**：后端开发 + 文档编写
- **应对时间窗口**：M1 Week 1-2（安装引导开发期）
- **应急方案**：增加安装 Token 机制（一次性随机 Token 仅在当前进程生命周期有效），即使 `/install` 对外开放，无 Token 也无法提交安装表单

#### RISK-07：浏览器兼容性测试

- **触发条件**：M2 测试阶段发现 Safari/Firefox 关键流程不可用
- **应对负责人**：前端开发
- **应对时间窗口**：M2 测试与打磨阶段
- **应急方案**：(1) 优先修复 Safari（iOS 用户占比高）；(2) Firefox 仅保证核心阅读流程可用，后台管理可降级为"推荐使用 Chrome"

### 9.4 风险跟踪

| 跟踪项 | 说明 |
|--------|------|
| **评审频率** | 每周进度回顾时同步评估风险状态变化 |
| **风险升级** | 当高等级风险（H）实际触发时，需在 24h 内制定应对方案并更新本文档 |
| **风险关闭** | 风险不再适用或已成功化解后，标记为"已关闭"并记录关闭原因 |
| **新增风险** | 开发过程中识别的新风险及时补充至 9.1 节，并同步更新风险矩阵 |
