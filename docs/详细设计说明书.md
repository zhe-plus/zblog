**依赖规则**：
- Controller 只能依赖 Service 和 DTO
- Service 只能依赖 Repository、其他 Service 和工具类

- Service 之间避免循环依赖

### 2.3 应用启动流程

```
Spring Boot 启动
  → 1. 加载 application.yml 配置
  → 2. 根据 spring.profiles.active 选择数据源（sqlite/mysql/pg）
  → 3. InstallDetectFilter 注册（检测安装状态）
     ├── 未安装：跳过 Flyway，所有请求重定向至 /install
     └── 已安装：继续下一步
  → 4. Flyway 自动执行数据库迁移脚本（仅已安装时执行）
  → 5. Spring Security 初始化认证授权配置
  → 6. 内嵌 Tomcat 启动
  → 7. 应用就绪
```

> **关键顺序**：安装检测必须在 Flyway 之前。若数据库未配置（未安装状态），Flyway 迁移会因无法连接数据库而失败。实现方案：`InstallDetectFilter` 在 `FlywayAutoConfiguration` 之前通过 `@AutoConfigureBefore` 或 `FilterRegistrationBean.setOrder()` 确保先执行；未安装时通过条件配置禁用 Flyway 自动执行。

**安装状态检测流程（InstallDetectFilter）**：

```
请求到达 → InstallDetectFilter
    │
    ├── 已安装（.zblog_installed 标记存在 或 settings 表有数据）
    │   ├── 路径 = /install → 返回 404
    │   └── 路径 ≠ /install → 正常处理
    │
    └── 未安装
        ├── 路径 = /install 或 /api/v1/install/* 或 /static/** → 正常处理
        └── 其他所有路径 → 重定向到 /install
```

> **安装引导页面实现约束**：`InstallController` 不注入任何 Repository 或 Service Bean，仅返回静态 Thymeleaf 模板（`install/step1-database.html`、`step2-account.html`、`step3-site.html`）。安装引导所需的 CSS/JS 静态资源路径（`/css/**`、`/js/**`、`/images/**`）已在 `InstallDetectFilter` 白名单中放行。安装引导页面不依赖数据库连接，确保在数据库未配置时也能正常渲染。

### 2.4 请求处理管道

```
HTTP 请求
  → Tomcat Connector
  → InstallDetectFilter（安装状态检测）
  → Spring Security Filter Chain（认证/授权/CSRF）
  → DispatcherServlet
  → HandlerMapping（路由匹配）
  → Controller 方法
  → Service 层（业务逻辑）
  → Repository 层（数据访问）
  → 数据库 / 文件系统
```

```sql
    content        TEXT    NOT NULL,
    version_number INTEGER NOT NULL,
    change_summary TEXT,
    created_at     TEXT    NOT NULL DEFAULT (datetime('now'))

CREATE TABLE IF NOT EXISTS links (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    name        TEXT    NOT NULL,
    url         TEXT    NOT NULL,
    description TEXT,
    sort_order  INTEGER NOT NULL DEFAULT 0,
    is_visible  INTEGER NOT NULL DEFAULT 1,
    created_at  TEXT    NOT NULL DEFAULT (datetime('now')),
    updated_at  TEXT    NOT NULL DEFAULT (datetime('now'))

-- 表 13：audit_logs（操作日志表）
-- v1.0 为单用户系统，user_id 可为 NULL；v1.2+ 多用户时补充外键约束
CREATE TABLE IF NOT EXISTS audit_logs (
    id          INTEGER PRIMARY KEY AUTOINCREMENT,
    action      TEXT NOT NULL,
    target_type TEXT,
    target_id   INTEGER,
    detail      TEXT,
    ip_address  TEXT,
    user_id     INTEGER REFERENCES users(id),
    created_at  TEXT NOT NULL DEFAULT (datetime('now'))
);
CREATE INDEX idx_audit_logs_action ON audit_logs(action);
CREATE INDEX idx_audit_logs_created ON audit_logs(created_at);

-- FTS5 全文搜索虚拟表（SQLite 专用）
CREATE VIRTUAL TABLE IF NOT EXISTS posts_fts USING fts5(
    title,
    content,
    content='posts',
    content_rowid='id'
);

-- FTS 同步触发器
CREATE TRIGGER IF NOT EXISTS posts_fts_insert AFTER INSERT ON posts BEGIN
    INSERT INTO posts_fts(rowid, title, content) VALUES (new.id, new.title, new.content);
END;
CREATE TRIGGER IF NOT EXISTS posts_fts_update AFTER UPDATE ON posts BEGIN
    UPDATE posts_fts SET title = new.title, content = new.content WHERE rowid = old.id;
END;
CREATE TRIGGER IF NOT EXISTS posts_fts_delete AFTER DELETE ON posts BEGIN
    DELETE FROM posts_fts WHERE rowid = old.id;
END;
-- 注意：软删除（设置 deleted_at）不会触发 AFTER DELETE 触发器。
-- 搜索查询中已包含 deleted_at IS NULL 条件，功能上不会泄露已删除文章。
-- 物理删除（实际 DELETE）时才从 FTS 索引中清理，v1.0 可接受此冗余，
-- v1.1+ 建议增加 AFTER UPDATE 触发器在软删除时同步清理 FTS 索引。
```

### 3.2 MySQL / PostgreSQL DDL 差异

| 差异项 | SQLite | MySQL | PostgreSQL |
|--------|--------|-------|------------|
| 主键自增 | `INTEGER PRIMARY KEY AUTOINCREMENT` | `BIGINT AUTO_INCREMENT PRIMARY KEY` | `BIGSERIAL PRIMARY KEY` |
| 时间默认值 | `datetime('now')` | `CURRENT_TIMESTAMP` | `NOW()` |
| 布尔值 | `INTEGER 0/1` | `TINYINT(1)` | `BOOLEAN` |
| 全文搜索 | FTS5 虚拟表+触发器 | `FULLTEXT INDEX` on posts(title, content) | `tsvector` 列 + GIN 索引 |
| TEXT | TEXT | LONGTEXT (content 字段) | TEXT |

### 3.4 JPA 实体示例

#### 3.4.1 Post 实体

```java
@Entity
@Table(name = "posts", indexes = {
    @Index(name = "idx_posts_status", columnList = "status"),
    @Index(name = "idx_posts_slug", columnList = "slug"),
    @Index(name = "idx_posts_category", columnList = "category_id"),
    @Index(name = "idx_posts_published", columnList = "publishedAt"),
    @Index(name = "idx_posts_deleted", columnList = "deleted_at"),
    @Index(name = "idx_posts_status_published", columnList = "status, publishedAt")
})
public class Post {
    @Id @GeneratedValue(strategy = GenerationType.IDENTITY)
    private Long id;

    @Column(nullable = false)
    private String title;

    @Column(nullable = false, unique = true)
    private String slug;

    @Column(nullable = false, columnDefinition = "TEXT")
    private String content;

    @Column(name = "content_html", nullable = false, columnDefinition = "TEXT")
    private String contentHtml;

    @Column(columnDefinition = "TEXT")
    private String summary;

    @Enumerated(EnumType.STRING)
    @Column(nullable = false, length = 20)
    private PostStatus status = PostStatus.DRAFT;

    @ManyToOne(fetch = FetchType.LAZY)
    @JoinColumn(name = "category_id")
    private Category category;

    @Column(name = "is_pinned", nullable = false)
    private Boolean isPinned = false;

    @Column(name = "published_at")
    private LocalDateTime publishedAt;

    @Column(name = "scheduled_at")
    private LocalDateTime scheduledAt;

    @Column(name = "created_at", nullable = false, updatable = false)
    private LocalDateTime createdAt;

    @Column(name = "updated_at", nullable = false)
    private LocalDateTime updatedAt;

    @Column(name = "deleted_at")
    private LocalDateTime deletedAt;

    @ManyToMany
    @JoinTable(name = "post_tags",
        joinColumns = @JoinColumn(name = "post_id"),
        inverseJoinColumns = @JoinColumn(name = "tag_id"))
    private Set<Tag> tags = new HashSet<>();

    @OneToMany(mappedBy = "post", cascade = CascadeType.ALL, orphanRemoval = true)
    @OrderBy("versionNumber DESC")
    private List<PostVersion> versions = new ArrayList<>();

    @PrePersist
    protected void onCreate() {
        this.createdAt = LocalDateTime.now();
        this.updatedAt = LocalDateTime.now();
        if (this.status == PostStatus.PUBLIC || this.status == PostStatus.PRIVATE) {
            this.publishedAt = LocalDateTime.now();
        }
    }

    @PreUpdate
    protected void onUpdate() {
        this.updatedAt = LocalDateTime.now();
        // 注意：publishedAt 的设置由 Service 层控制，不由 @PreUpdate 自动处理。
        // 当 PostService.changeStatus() 将状态从 draft 变为 public/private 时，
        // 若 publishedAt 为 null 则设置为当前时间；状态变为 draft 时保留原值。
    }
    // getters/setters ...
}
```

### 3.5 索引设计汇总

| 索引 | 表 | 字段 | 类型 | 理由 |
|------|-----|------|------|------|

### 4.5 安装引导 API

> **状态管理说明**：安装引导采用"暂存→统一提交"模式。`step1` 和 `step2` 仅将数据暂存于服务端 Session（`HttpSession`），不执行任何持久化操作。`step3` 统一执行所有持久化操作：创建数据库表（Flyway 迁移）→ 创建管理员用户（bcrypt 密码哈希）→ 写入站点设置 → 写入 `.zblog_installed` 标记文件。此模式确保安装过程的原子性——若中途中断，不会留下不完整的数据。
>
> **step3 Flyway 执行技术方案**：由于未安装状态下 Spring Boot 的 `FlywayAutoConfiguration` 默认会在 `ApplicationContext` 刷新时自动执行迁移，而安装引导阶段数据库尚未配置，因此需要以下技术方案确保 Flyway 在正确的时机执行：
> 1. **禁用自动执行**：在 `application.yml` 中设置 `spring.flyway.enabled=false`，阻止 `FlywayAutoConfiguration` 在启动时自动执行迁移
> 2. **手动注册 Flyway Bean**：通过 `FlywayConfig` 配置类（`@Configuration`）手动创建 `Flyway` Bean，配置数据源但不自动执行 `migrate()`
> 3. **step3 手动调用**：`InstallService` 中注入 `Flyway` Bean，在 step3 执行持久化时调用 `flyway.migrate()`，迁移成功后在同一事务中继续创建用户和写入设置

### 5.2 Markdown 渲染与安全清洗流程

```java
@Service
public class MarkdownService {
    private final Parser parser;        // CommonMark 解析器（含 GFM 扩展）
    private final HtmlRenderer renderer;
    private final BacklinkService backlinkService;

    public String renderToHtml(String markdown, Long postId) {
        // 1. 解析双向链接
        String processed = postId != null
            ? backlinkService.resolveWikiLinks(markdown, postId)
            : markdown;
        // 2. Markdown → HTML
        Node document = parser.parse(processed);
        String html = renderer.render(document);
        // 3. HTML 清洗
        return HtmlSanitizer.sanitize(html);
    }

    public String generateSummary(String markdown, int maxLength) {
        // 去除 Markdown 标记，提取纯文本，截取 maxLength 字符
        String text = markdown.replaceAll("#+\\s+", "")
            .replaceAll("\\*\\*(.+?)\\*\\*", "$1")
            .replaceAll("`{1,3}[^`]*`{1,3}", "")
            .replaceAll("!\\[.*?\\]\\(.*?\\)", "")
            .replaceAll("\\[([^\\]]+)\\]\\(.*?\\)", "$1")
            .replaceAll("\\n+", " ").trim();
        // 解码 HTML 实体（如 &amp; → &, &lt; → <），避免截取后出现乱码
        text = HtmlSanitizer.decodeEntities(text);
        return text.length() <= maxLength ? text : text.substring(0, maxLength) + "...";
    }
}
```

### 5.3 双向链接解析算法

```
正文中的 [[other-article-slug]]
  → 1. 正则匹配 [[...]] 提取 slug 列表
  → 2. 批量查询 posts WHERE slug IN (...) AND status='public' AND deleted_at IS NULL
  → 3. 存在 → 替换为 [title](/post/slug)；不存在 → 保留原文本 [slug](/post/slug)
```

```java
@Service
public class BacklinkService {
    private static final Pattern WIKI_LINK = Pattern.compile("\\[\\[([^\\]]+)\\]\\]");

    /**
     * 解析 Markdown 中的 [[slug]] 双向链接，替换为 Markdown 链接语法。
     * 
     * 性能说明：为避免在 MarkdownService.renderToHtml() 渲染流程中产生 N+1 查询
     * （批量渲染场景如 Sitemap/RSS 生成遍历所有文章时尤为明显），此方法先收集当前
     * Markdown 中所有 [[slug]] 引用，然后通过一次批量 IN 查询获取所有匹配文章的
     * slug→title 映射到内存 Map，渲染时直接查 Map 而非逐条查库。
     */
    public String resolveWikiLinks(String markdown, Long currentPostId) {
        Matcher m = WIKI_LINK.matcher(markdown);
        
        // 1. 收集所有引用的 slug
        Set<String> slugs = new HashSet<>();
        while (m.find()) {
            slugs.add(m.group(1).trim());
        }
        
        // 2. 批量预加载 slug→title 映射（避免 N+1 查询）
        Map<String, String> slugTitleMap;
        if (!slugs.isEmpty()) {
            slugTitleMap = postRepository
                .findBySlugInAndStatusAndDeletedAtIsNull(slugs, PostStatus.PUBLIC)
                .stream()
                .collect(Collectors.toMap(Post::getSlug, Post::getTitle));
        } else {
            slugTitleMap = Map.of();
        }
        
        // 3. 替换所有匹配的 [[slug]]
        m.reset();
        StringBuffer sb = new StringBuffer();
        while (m.find()) {
            String slug = m.group(1).trim();
            String title = slugTitleMap.getOrDefault(slug, slug);
            String replacement = String.format("[%s](/post/%s)", title, slug);
            m.appendReplacement(sb, Matcher.quoteReplacement(replacement));
        }
        m.appendTail(sb);
        return sb.toString();
    }
}
```

### 5.5 Slug 生成算法

> **中文标题处理策略**：纯中文标题经转换后会全部变为连字符，最终 fallback 为 `"untitled"`。为避免多篇中文文章 Slug 冲突，当生成的 Slug 为空字符串或与已有 Slug 冲突时，自动追加递增数字后缀（如 `untitled-1`、`untitled-2`）。v1.1+ 建议引入拼音转换（pinyin4j）生成更语义化的 Slug。

```java
public class SlugUtils {
    private static final Set<String> RESERVED = Set.of(
        "search", "archive", "rss.xml", "sitemap.xml", "robots.txt",
        "feed", "admin", "api", "install", "login", "logout",
        "page", "post", "category", "tag", "uploads", "static", "health"
    );

    public static String generate(String title) {
        String slug = title.toLowerCase().trim()
            .replaceAll("[^a-z0-9\\u4e00-\\u9fff\\-\\s]", "")
            .replaceAll("[\\u4e00-\\u9fff]", "-")  // 中文→连字符
            .replaceAll("\\s+", "-")
            .replaceAll("-{2,}", "-")
            .replaceAll("^-|-$", "");
        return slug.isEmpty() ? "untitled" : slug;
    }

    /** 确保唯一性：在 Repository 层调用，Slug 冲突时追加数字后缀 */
    public static String ensureUnique(String baseSlug, Function<String, Boolean> exists) {
        if (!exists.apply(baseSlug)) return baseSlug;
        int suffix = 1;
        String candidate;
        do { candidate = baseSlug + "-" + (suffix++); }
        while (exists.apply(candidate));
        return candidate;
    }

    public static boolean isReserved(String slug) {
        return RESERVED.contains(slug.toLowerCase());
    }
}
```

### 5.7 页面缓存服务（NFR-P10 P0，v1.0 实现）

```java
@Service
public class CacheService {
    private final Cache<String, String> pageCache = Caffeine.newBuilder()
        .maximumSize(500)
        .expireAfterWrite(30, TimeUnit.MINUTES)
        .build();

    public Optional<String> get(String key) {
        return Optional.ofNullable(pageCache.getIfPresent(key));
    }

    public void put(String key, String html) {
        pageCache.put(key, html);
    }

    public void evict(String key) {
        pageCache.invalidate(key);
    }

    /**
     * 文章变更时清除相关缓存：单篇文章、首页、分类页、标签页、归档页。
     * @param tagSlugs 文章关联的标签 slug 集合，用于逐一清除标签页缓存
     */
    public void evictPostRelated(String postSlug, String categorySlug, Set<String> tagSlugs) {
        evict("post:" + postSlug);
        evict("home:page:1");
        if (categorySlug != null) {
            evict("category:" + categorySlug + ":page:1");
        }
        evict("archive");
        // 标签页缓存：Caffeine 不支持通配符 key 清除，需逐一清除
        if (tagSlugs != null) {
            tagSlugs.forEach(slug -> evict("tag:" + slug + ":page:1"));
        }
    }
}
```

> v1.0 使用 Caffeine JVM 本地缓存，满足 NFR-P10（P0）页面缓存要求。缓存键设计：`post:{slug}`、`home:page:{n}`、`category:{slug}:page:{n}`、`tag:{slug}:page:{n}`、`archive`、`search:{q}:page:{n}`。
> 
> **标签缓存清除说明**：Caffeine 不支持通配符 key 清除（如 `tag:*`），因此在 `evictPostRelated()` 方法参数中传入文章关联的标签 slug 集合（`Set<String> tagSlugs`），逐一清除对应标签页缓存。简化方案：标签页缓存 key 可设计为 `tag:{slug}`（不分页），标签页每次请求时重新查询分页数据，仅缓存标签下的文章列表元数据。

### 6.2 登录保护（频率限制）

> **已知限制**：v1.0 使用内存 `ConcurrentHashMap` 存储登录失败计数，应用重启后计数清零。对于个人博客系统（单用户），攻击场景有限，此方案可接受。Docker 环境下容器重启较频繁，v1.1 评估将失败计数持久化到 `audit_logs` 或 `settings` 表。

```java
@Component
public class RateLimitFilter extends OncePerRequestFilter {
    // v1.0：内存存储，重启后清零。v1.1 评估持久化方案
    private final ConcurrentHashMap<String, Pair<Integer, LocalDateTime>> failures = new ConcurrentHashMap<>();
    private static final int MAX_ATTEMPTS = 5;
    private static final long LOCK_MINUTES = 15;

    @Override
    protected void doFilterInternal(HttpServletRequest req, HttpServletResponse res,
                                     FilterChain chain) throws IOException, ServletException {
        if (!isLoginRequest(req)) { chain.doFilter(req, res); return; }

        String ip = getClientIp(req);
        Pair<Integer, LocalDateTime> record = failures.get(ip);

        // 检查锁定状态
        if (record != null && record.getSecond() != null
            && LocalDateTime.now().isBefore(record.getSecond())) {
            sendLockedResponse(res);
            return;
        }

        ContentCachingResponseWrapper wrapper = new ContentCachingResponseWrapper(res);
        chain.doFilter(req, wrapper);

        // 处理登录结果
        if (wrapper.getStatus() == 401) {
            failures.compute(ip, (k, v) -> {
                int count = (v == null) ? 1 : v.getFirst() + 1;
                return Pair.of(count, count >= MAX_ATTEMPTS
                    ? LocalDateTime.now().plusMinutes(LOCK_MINUTES) : null);
            });
        } else if (wrapper.getStatus() == 200) {
            failures.remove(ip); // 登录成功清除记录
        }
        wrapper.copyBodyToResponse();
    }
}
```

### 6.3 XSS 防护策略

| 层级 | 防护措施 | 说明 |
|------|---------|------|
| 输入层 | `@Valid` Bean Validation | 参数类型和长度校验 |
| 渲染层 | Markdown → HTML → Jsoup Safelist | 白名单清洗，移除危险标签 |
| 输出层 | Thymeleaf `th:text` 自动转义 | 模板引擎默认 HTML 转义 |
| HTTP 头 | Content-Security-Policy | 限制脚本来源 |
| Cookie | HttpOnly + Secure + SameSite=Lax | 防止 JS 读取和 CSRF |

**HTML 清洗白名单**：允许 `h1-h6, p, br, strong, em, del, a(href), img(src,alt), ul/ol/li, blockquote, pre, code(class), table/tr/td/th, span(class)`，禁止 `<script>`、`<iframe>`、`<object>`、`on*` 事件属性。

### 6.4 密码安全

- 存储：bcrypt(12) 哈希
- 强度：≥ 8 位，含字母和数字
- 修改密码：验证原密码 → 验证新密码强度 → bcrypt 加密 → 更新

## 7. 前台前端设计（Thymeleaf 模板）

### 7.0 前台路由与 PageController 设计

> **`/{slug}` 路由优先级设计**：独立页面通过 `/{slug}` 路径访问（如 `/about`、`/links`），该路径为通配路由，必须确保在所有系统保留路径之后匹配。

```java
@Controller
// Spring Boot 3.x 默认使用 PathPatternParser，具体路径优先于通配路径匹配
// @RequestMapping 中具体路径（如 /search）自动优先于 /{slug} 模式
public class PageController {
    
    @GetMapping("/{slug}")
    public String page(@PathVariable String slug, Model model) {
        // 1. 黑名单双重校验（防御性编程）
        if (SlugUtils.isReserved(slug)) {
            throw new ResponseStatusException(HttpStatus.NOT_FOUND);
        }
        // 2. 查询公开独立页面
        Page page = pageService.getBySlug(slug);
        if (page == null) {
            throw new ResponseStatusException(HttpStatus.NOT_FOUND);
        }
        // 3. 渲染对应模板
        model.addAttribute("page", page);
        return "page";
    }
}
```

**路由匹配优先级保证**：
- Spring Boot 3.x 默认使用 `PathPatternParser`（替代 AntPathMatcher），具体路径（如 `/search`、`/archive`）自动优先于模式路径（`/{slug}`）
- `/{slug}` 仅匹配单层路径（如 `/about`），不匹配多层路径（如 `/page/about`），与 Spring Security 中 `/*` 的语义一致
- Controller 内部通过 `SlugUtils.isReserved()` 做黑名单二次校验，双保险防止系统路由被独立页面覆盖
- 即使 `/{slug}` 意外匹配了保留路径，也会因黑名单校验返回 404

### 8.0 React SPA 路由的 Spring MVC 配置

> **`/admin/**` 路由处理**：React SPA 的 `index.html` 入口文件放置在 `src/main/resources/static/admin/index.html`，作为静态资源由 Spring Boot 自动 serve。对于 `/admin/posts/new` 等 SPA 子路由，需要配置 ViewController 将所有 `/admin/**` 请求 forward 到 `index.html`，由 React Router 接管前端路由。

```java
@Configuration
public class WebMvcConfig implements WebMvcConfigurer {
    
    @Override
    public void addViewControllers(ViewControllerRegistry registry) {
        // 所有 /admin/** 非 API 请求 forward 到 React SPA 入口
        registry.addViewController("/admin").setViewName("forward:/admin/index.html");
        registry.addViewController("/admin/").setViewName("forward:/admin/index.html");
        // 子路径转发：React Router 处理 /admin/posts、/admin/settings 等
        // 注意：此配置需放在 SecurityConfig 认证配置之后，
        // 确保 /api/v1/admin/** API 请求返回 JSON 401，而非 forward 到 index.html
    }
    
    @Override
    public void addResourceHandlers(ResourceHandlerRegistry registry) {
        // 确保 /admin/** 下的静态资源（JS/CSS/图片）能被正确 serve
        registry.addResourceHandler("/admin/**")
            .addResourceLocations("classpath:/static/admin/");
    }
}
```

**关键设计决策**：
- `index.html` 放在 `static/admin/`（而非 `templates/admin/`），作为纯静态资源，不经过 Thymeleaf 渲染
- `/admin/**` 子路径请求（如 `/admin/posts/new`）通过 Spring MVC `forward` 返回 `index.html`，由 React Router 接管前端路由
- **API vs SPA 请求区分**：Spring Security 中 `/api/v1/admin/**` 返回 JSON 401（未认证），`/admin/**`（非 API）返回 `index.html` 让 React 处理 401 重定向。两者通过 URL 前缀自然区分：API 路径含 `/api/`，SPA 页面路径不含 `/api/`
- Vite 构建产物（`dist/` 目录）全部拷贝到 `static/admin/`，`index.html` 中的 `<script src="/admin/assets/main.js">` 等引用能被 Spring Boot 正确识别为静态资源

### 8.4 API 客户端封装

```typescript
// api/client.ts
class ApiClient {
    private csrfToken: string | null = null;

    private async request<T>(method: string, path: string, body?: unknown): Promise<T> {
        const headers: Record<string, string> = { 'Content-Type': 'application/json' };
        if (this.csrfToken) headers['X-CSRF-Token'] = this.csrfToken;

        const res = await fetch(path, { method, headers, credentials: 'same-origin',
            body: body ? JSON.stringify(body) : undefined });

        if (res.status === 401) { window.location.href = '/admin/login'; throw new Error('Unauthorized'); }

        const csrf = res.headers.get('X-CSRF-Token');
        if (csrf) this.csrfToken = csrf;

        if (!res.ok) { const err = await res.json(); throw new ApiException(err.error.code, err.error.message, res.status); }
        if (res.status === 204) return undefined as T;
        return res.json();
    }

    get<T>(path: string) { return this.request<T>('GET', path); }
    post<T>(path: string, body?: unknown) { return this.request<T>('POST', path, body); }
    put<T>(path: string, body?: unknown) { return this.request<T>('PUT', path, body); }
    patch<T>(path: string, body?: unknown) { return this.request<T>('PATCH', path, body); }
    delete<T>(path: string) { return this.request<T>('DELETE', path); }

    async upload<T>(path: string, file: File): Promise<T> {
        const fd = new FormData(); fd.append('file', file);
        const res = await fetch(path, { method: 'POST', credentials: 'same-origin',
            headers: this.csrfToken ? { 'X-CSRF-Token': this.csrfToken } : {}, body: fd });
        if (res.status === 401) { window.location.href = '/admin/login'; throw new Error('Unauthorized'); }
        if (!res.ok) { const err = await res.json(); throw new ApiException(err.error.code, err.error.message, res.status); }
        return res.json();
    }
}
export const api = new ApiClient();
```

### 9.3 Dockerfile（多阶段构建）

> **前端资源路径说明**：前端构建产物应打包进 Spring Boot Fat JAR 的 classpath 中（即 `src/main/resources/static/admin/`），而非运行时外部目录。Dockerfile 中后端构建阶段通过 Gradle 插件（如 `com.github.node-gradle`）自动引入前端构建产物到 resources 目录。下方 Dockerfile 展示标准多阶段构建流程。
>
> **COPY 路径确认**：
> - 阶段 2 的工作目录为 `/app`，`COPY backend/ .` 将后端源码拷贝到 `/app/` 下（含 `src/main/resources/` 目录）
> - `COPY --from=frontend-build /app/dist/ src/main/resources/static/admin/` 将前端产物拷贝到 `/app/src/main/resources/static/admin/`，Gradle `bootJar` 打包时会将 `src/main/resources/` 下的文件打入 JAR classpath
> - **关键前置条件**：`COPY backend/ .` 必须先于 `COPY --from=frontend-build` 执行，确保 `src/main/resources/` 目录已存在
> - 建议在 backend 的 `src/main/resources/static/admin/` 中放置 `.gitkeep` 占位文件，确保目录在 Git 中已存在，避免 `COPY` 因目标目录不存在而失败
> - 前端 Vite 构建产物 `dist/` 目录结构应包含 `index.html` 和 `assets/` 子目录（JS/CSS 等），与 `index.html` 中 `<script src="/admin/assets/main.js">` 的引用路径一致

```bash
mkdir -p "$BACKUP_DIR"

# 备份数据库
sqlite3 /opt/zblog/data/zblog.db ".backup '$BACKUP_DIR/zblog_$DATE.db'"

# 备份上传文件
tar -czf "$BACKUP_DIR/uploads_$DATE.tar.gz" -C /opt/zblog/data uploads/

# 清理 30 天前的旧备份
find "$BACKUP_DIR" -mtime +30 -delete
```

crontab：`0 3 * * * /opt/zblog/scripts/backup.sh >> /var/log/zblog_backup.log 2>&1`

**恢复流程**：停止服务 → 覆盖 `data/zblog.db` → 解压 `uploads/` → 启动服务 → 验证

### 9.6 部署步骤总结

**方案 A（systemd）**：
