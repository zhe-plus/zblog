## 二、阻塞性问题（必须修正，否则影响开发）

### 1. 前台搜索 API 路径不一致

- **需求文档** 5.4.2 定义前台搜索为 `GET /api/v1/search?q=keyword`
- **详细设计** 4.3.6 也定义为 `GET /api/v1/search?q=keyword`
- 但 **SecurityConfig** (6.1) 中 CSRF 排除路径写的是 `/api/v1/search/**`，而授权 permitAll 写的是 `/api/v1/search/**`
- 前台搜索本身是 `GET` 请求，CSRF 排除是多余的（GET 请求不触发 CSRF），但不算错误
- **真正的问题**：SecurityConfig 中 permitAll 路径写了 `/*`（单星号），意图匹配 `/{slug}` 独立页面，但 `/*` 在 Spring Security 中只匹配单层路径。如果页面 slug 为 `about`，可以匹配；但这会与 `/rss.xml`、`/sitemap.xml`、`/robots.txt` 等已单独列出的路径产生重复匹配，且 `/*` 不匹配 `/admin/login` 等。建议明确注释 `/*` 仅用于顶层独立页面，或改用更精确的路由策略。

**修正意见**：SecurityConfig 中 `/*` 的用途需要注释说明，或替换为自定义 `RequestMatcher` 逻辑，避免开发时产生歧义。

---

### 2. 安装引导步骤3的 API 设计与流程描述不一致

- 需求文档 6.1.2 描述安装引导为 **3 步**：(1) 数据库配置 → (2) 管理员账号 → (3) 站点信息
- 详细设计 4.5 定义了 4 个安装 API：`test-connection`、`step1`、`step2`、`step3`
- **问题**：`step3` 的描述为"完成安装（建表+创建用户+站点信息）"，但 `step2` 已经创建了管理员账号。那么 `step3` 是否重复创建用户？还是 `step2` 只是保存账号信息到临时 Session，`step3` 统一执行持久化？

**修正意见**：明确安装引导的状态管理——各 step 是独立持久化还是 step3 统一提交。建议方案：`step1` 保存数据库配置到临时 Session，`step2` 保存管理员信息到临时 Session，`step3` 统一执行（建表 + 创建用户 + 写入站点设置 + 写入锁定标记）。当前描述含糊，开发时会有歧义。

### 3. `posts.content_html` 字段 NOT NULL 但默认空字符串，与安装引导流程冲突

- DDL 中 `content_html TEXT NOT NULL DEFAULT ''`
- 但 JPA 实体 `Post.contentHtml` 标注了 `@Column(nullable = false)`
- **问题**：安装引导阶段（未安装状态）Flyway 尚未执行，数据库表不存在。但 `step3` 才执行建表。如果在 `step1` 和 `step2` 之间用户通过浏览器进行其他操作，所有请求被重定向到 `/install`，这部分是 OK 的。但需确认：安装引导页面本身是否依赖数据库？需求文档说"安装引导页面使用独立的简洁模板，不依赖数据库"，这个设计是正确的，但详细设计未说明安装引导页面的 Controller 如何绕过 `InstallDetectFilter` 的数据库依赖检测。

**修正意见**：补充 `InstallController` 的实现说明——它不依赖任何 Repository/Service，仅返回静态 Thymeleaf 模板，且静态资源路径（CSS/JS）需在 `InstallDetectFilter` 的放行白名单中明确列出。

### 4. SQLite FTS5 触发器在软删除场景下有数据一致性风险

DDL 中的 FTS5 触发器：

```sql
CREATE TRIGGER posts_fts_delete AFTER DELETE ON posts BEGIN
    DELETE FROM posts_fts WHERE rowid = old.id;
END;
```

- **问题**：zblog 使用软删除（`deleted_at`），不会触发 `AFTER DELETE`。软删除的文章仍保留在 `posts_fts` 中，前台搜索会命中已软删除的文章（虽然查询中有 `deleted_at IS NULL` 过滤，但搜索索引中存在冗余数据，影响搜索结果排序和性能）。
- **修正意见**：搜索查询 SQL 中已有 `p.deleted_at IS NULL` 条件，功能上不会泄露已删除文章。但建议补充说明：软删除后 FTS 索引不主动清理，仅在物理删除时清理。或在文章软删除时增加一个 `AFTER UPDATE` 触发器，当 `deleted_at` 从 NULL 变为非 NULL 时同步从 FTS 中删除。

## 三、重要问题（应修正，影响开发效率）

### 5. `comments` 表在 v1.0 建表但评论提交 API (`POST /api/v1/comments`) 在前台公开 API 中定义且无版本标注

- 详细设计 4.3.8 定义了 `POST /api/v1/comments` 评论提交 API，包含完整的请求体和字段约束
- 但评论系统整体在 **v1.1** 启用（4.4.5 评论管理 API 标注了 v1.1）
- **问题**：前台评论提交 API 未标注版本。如果 v1.0 不实现评论功能，那么 `POST /api/v1/comments` 在 v1.0 中应该返回 404 或 501。但 SecurityConfig 中已将 `/api/v1/comments` 加入 permitAll 路径。
- **修正意见**：明确 `POST /api/v1/comments` 的版本归属。如果 v1.0 不启用，应在 SecurityConfig 中排除该路径或标注 v1.1 启用。前台文章详情页模板中也不应渲染评论表单。

---

### 6. 前台文章详情页 `GET /api/v1/posts/{slug}` 返回了 `content`（Markdown 原文），但前台是 Thymeleaf 服务端渲染

- 详细设计 4.3.2 的响应体包含 `content`（Markdown 原文）和 `content_html`（渲染后 HTML）
- **问题**：前台文章详情页由 Thymeleaf 服务端渲染，直接使用 `content_html` 输出。前台公开 API 返回 `content` 原文没有必要，且暴露 Markdown 源码可能不符合预期（虽然不算安全问题）。
- **修正意见**：前台公开 API 的文章详情响应可不返回 `content` 字段，或标注该 API 主要供未来前台 React 组件使用。当前 Thymeleaf 模板直接从 Service 层获取数据，不走 API。

### 7. `SlugUtils.generate()` 对中文标题的处理可能导致空 Slug

```java
.replaceAll("[^a-z0-9\\u4e00-\\u9fff\\-\\s]", "")
.replaceAll("[\\u4e00-\\u9fff]", "-")  // 中文→连字符
```

- **问题**：如果标题全是中文（如"我的第一篇文章"），第一步保留中文字符，第二步将中文全部替换为连字符 `-`，结果为 `------`，经过 `-{2,}` 合并为 `-`，再去除首尾 `-`，结果为空字符串，返回 `"untitled"`。多篇全中文标题文章都会生成 `untitled`，导致 Slug 冲突。
- **修正意见**：对中文标题的 Slug 生成策略需要改进。方案：(1) 使用拼音转换（引入 pinyin4j）；(2) 使用文章 ID 作为 Slug fallback（如 `post-1`）；(3) 使用时间戳。建议 v1.0 采用方案 (2)，在 Slug 为空或冲突时自动追加 ID。

### 8. 需求文档 NFR 汇总表引用了不存在的"第 6.3 节"监控编号

- 需求文档 4.6 非功能需求汇总表中有一行 `6.3 监控与告警`，引用了 NFR-O01~NFR-O06
- 但这些 NFR 编号以 `O` (Operations) 开头，与前面 NFR-P/S/A/M/SE 的前缀体系不同
- **问题**：4.6 汇总表中将 `6.3 监控与告警` 与 `4.1~4.5` 并列，但 6.3 是部署运维章节的内容，不属于第 4 章非功能需求。这导致汇总表结构不一致。
- **修正意见**：将 NFR-O 系列要么移入第 4 章作为 4.7 节，要么从 4.6 汇总表中移除，仅保留在第 6.3 节中。当前写法导致"非功能需求 73 条"的计数包含了运维监控需求，但这些需求优先级为 P2/P3，不影响 MVP。

### 9. 详细设计缺少 `Category` 和 `Tag` 等其他 JPA 实体示例

- 详细设计 3.4 仅给出了 `Post` 实体的完整 JPA 代码
- **问题**：`Category`、`Tag`、`Media`、`Comment`、`Page`、`Settings` 等实体缺少设计，开发时需要自行推断关联关系、索引注解等。
- **修正意见**：至少补充 `Category`、`Tag`、`Settings` 的实体定义（因为它们涉及不同的主键策略和关联模式），其余可标注"参照 Post 实体模式"。

### 10. `post_media` 表缺少 `ON DELETE CASCADE` 在 DDL 中一致

- 需求文档 5.3.2 中 `post_media` 表的外键约束未指定 `ON DELETE`
- 详细设计 DDL 中 `post_media` 指定了 `ON DELETE CASCADE`
- **问题**：两处不一致。需求文档说"用于追踪文章引用了哪些媒体文件"，CASCADE 删除意味着删除文章时自动删除引用关系，这是合理的。但需求文档未明确。
- **修正意见**：以详细设计 DDL 为准（CASCADE），在需求文档中补充说明。

## 四、建议性问题（建议修正，提升质量）

### 11. `audit_logs` 表缺少 `user_id` 字段

- 需求文档 ER 关系图标注 `users (1) ─────────── (N) audit_logs`，但 `audit_logs` 表结构中没有 `user_id` 外键字段
- **修正意见**：补充 `user_id INTEGER REFERENCES users(id)` 字段，或在 ER 图中移除该关联（v1.0 为单用户系统，可从 IP 或 action 中推断操作者）。

---

### 12. `settings` 表使用 `key` 作为主键，但 `key` 是 SQL 保留字

- SQLite 中 `key` 不是保留字，但 MySQL 中 `KEY` 是保留字
- **修正意见**：将字段名改为 `setting_key` 或使用反引号/引号转义，确保多数据库兼容。

---

### 13. `pages` 表缺少 `deleted_at` 的索引

- `posts` 表有 `idx_posts_deleted` 索引，但 `pages` 表没有
- **修正意见**：补充 `CREATE INDEX idx_pages_deleted ON pages(deleted_at)`，因为 pages 同样使用软删除。

---

### 14. MarkdownService 中 `generateSummary` 使用正则去除 Markdown 标记，但未处理 HTML 实体

- 如果 Markdown 内容中包含 `&amp;`、`&lt;` 等 HTML 实体，摘要中会保留这些实体
- **修正意见**：在截取后进行一次 HTML 实体反转义，或使用 commonmark-java 的 AST 遍历提取纯文本。

---

### 15. 前台 Thymeleaf 模板缺少 500 错误页

- 详细设计 10.2 审查记录中已发现并标注"v1.1 补充"
- **修正意见**：v1.0 应至少有一个简单的 500 错误页，避免 Spring Boot 默认 Whitelabel Error Page 暴露框架信息（违反 NFR-S08）。建议 v1.0 即提供基础 `500.html`。

---

### 16. 前台公开 API 返回的 JSON 字段命名不一致

- 需求文档 5.4.1 规定 JSON 字段使用 camelCase
- 但实际响应中混用了 snake_case（`created_at`、`published_at`、`page_size`、`is_pinned`、`content_html`）和 camelCase（`readingTime` 在设计意图中是 camelCase，但实际返回 `reading_time`）
- **修正意见**：统一决策——要么全 snake_case（通过 Jackson 全局配置 `PropertyNamingStrategies.SNAKE_CASE`），要么全 camelCase。当前实际实现倾向于 snake_case，建议在 5.4.1 中修正说明为"JSON 字段使用 snake_case，与数据库字段命名保持一致"。

---

### 17. Dockerfile 中前端构建产物路径与需求文档不一致

- 需求文档 6.1.5 Dockerfile：`COPY --from=frontend-build /app/dist static/admin/`
- 详细设计 9.3 Dockerfile：`COPY --from=frontend-build /app/dist /app/static/admin/`
- 需求文档 5.2.3 目录结构中前端构建产物应拷贝到 `src/main/resources/static/admin/`
- **问题**：三个位置的路径描述不完全一致。Spring Boot 静态资源应在 classpath:`static/` 下，但 Dockerfile 直接拷贝到 `/app/static/admin/` 可能不被 Spring Boot 识别为静态资源（除非配置了额外的静态资源路径）。
- **修正意见**：明确前端构建产物在 Fat JAR 中的位置。如果是构建时拷贝到 `src/main/resources/static/admin/` 然后打包进 JAR，则 Dockerfile 后端构建阶段就应包含前端资源。如果是运行时外部目录，则需配置 `spring.web.resources.static-locations`。建议统一为：后端构建阶段引入前端构建产物。
