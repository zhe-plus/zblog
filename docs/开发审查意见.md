## 三、重要问题（应修正，影响开发效率或质量）

### I-1. settings 表 DDL 中字段名 `value` 与需求文档 `setting_value` 不一致

- **需求文档** 5.3.2 表 10 中 settings 表字段命名为 `setting_value`（明确说明"字段命名为 `setting_value` 而非 `value`，避免与 MySQL 保留字冲突"）
- **详细设计** 3.1 DDL 脚本中 settings 表定义为 `value TEXT NOT NULL`（第 314 行），使用了 `value` 而非 `setting_value`
- **详细设计** 3.4.4 Settings JPA 实体中 `@Column(nullable = false, columnDefinition = "TEXT") private String value;`（第 570 行），字段名也为 `value`
- **问题**：`value` 在某些 SQL 上下文中可能成为保留字或引起歧义。需求文档已明确为解决此问题将字段命名为 `setting_value`，但详细设计的 DDL 和 JPA 实体未同步

**修正意见**：将详细设计 DDL 中 `value TEXT NOT NULL` 改为 `setting_value TEXT NOT NULL`；JPA 实体中 `@Column(name = "setting_value") private String value;`。确保与需求文档一致。

---

### I-2. Post 实体 `@Table` 索引注解缺少 `idx_posts_category` 和 `idx_posts_deleted`

- **详细设计** 3.1 DDL 中定义了 6 个 posts 索引：`idx_posts_status`、`idx_posts_slug`、`idx_posts_category`、`idx_posts_published`、`idx_posts_deleted`、`idx_posts_status_published`
- **详细设计** 3.4.1 Post 实体 `@Table(indexes = {...})` 仅声明了 4 个索引：`idx_posts_status`、`idx_posts_slug`、`idx_posts_published`、`idx_posts_status_published`（第 411-416 行）
- **问题**：缺少 `idx_posts_category`（category_id）和 `idx_posts_deleted`（deleted_at）索引的 JPA 注解声明。虽然 dev 环境使用 `ddl-auto: update` 时 Hibernate 会自动建表，但生产环境使用 Flyway DDL 脚本时不受影响——不过 JPA 实体注解与 DDL 不一致会导致混淆

**修正意见**：在 Post 实体 `@Table` 注解中补充：
```java
@Index(name = "idx_posts_category", columnList = "category_id"),
@Index(name = "idx_posts_deleted", columnList = "deleted_at")
```
使 JPA 实体注解与 DDL 脚本完全一致。

---

### I-3. Dockerfile 缺少 `.gitkeep` 占位文件的实际创建步骤

- **详细设计** 9.3 中建议"在 backend 的 `src/main/resources/static/admin/` 中放置 `.gitkeep` 占位文件，确保目录在 Git 中已存在"
- **问题**：这是设计建议而非实际操作步骤。如果开发者未手动创建 `.gitkeep` 文件，`COPY --from=frontend-build /app/dist src/main/resources/static/admin/` 在 `src/main/resources/static/admin/` 目录不存在时，Docker COPY 会自动创建目录——但此处 COPY 目标是相对路径 `src/main/resources/static/admin/`，Docker 会创建为文件而非目录（当目标路径不以 `/` 结尾时），导致构建失败
- **修正意见**：(1) 在文档中明确开发者需在 backend 源码中创建 `src/main/resources/static/admin/.gitkeep` 文件并提交到 Git；(2) 或将 COPY 目标路径改为以 `/` 结尾：`COPY --from=frontend-build /app/dist/ src/main/resources/static/admin/`，确保 Docker 将其视为目录

## 四、建议性问题（建议修正，提升质量）

### S-1. WebMvcConfig 中 `/admin/**` 子路径 forward 未涵盖深层嵌套路由

- **详细设计** 8.0 中 `WebMvcConfig.addViewControllers()` 仅注册了 `/admin` 和 `/admin/` 两个路径的 forward
- **问题**：`/admin/posts/new`、`/admin/posts/1/edit` 等深层子路径的 forward 未在 `addViewControllers` 中注册。文档注释提到"子路径转发由 React Router 处理"，但 Spring MVC 中如果 `/admin/posts/new` 没有对应的 Controller 映射且没有 view controller forward，Spring Boot 会返回 404 而非 forward 到 `index.html`
- **修正意见**：需补充一条 catch-all 规则。推荐方案：注册 `/admin/**` 通配 forward 到 `index.html`，但需注意排除 `/admin/api/**` 或确保 API 路径使用 `/api/v1/admin/**` 前缀（当前设计中 API 路径已使用 `/api/v1/admin/**`，不会与 `/admin/**` 冲突）。具体实现：
```java
// 在 addResourceHandlers 中配置 SPA fallback，或使用自定义 error view resolver
registry.addViewController("/admin/{path:[^\\.]+}").setViewName("forward:/admin/index.html");
registry.addViewController("/admin/{path:^(?!api).*$}/**").setViewName("forward:/admin/index.html");
```
建议在编码阶段验证 Spring Boot 3.x 对 `/admin/**` 通配 forward 的行为，确保 SPA 子路由正确返回 `index.html`。

---

### S-2. `Post.isPinned` 字段类型为 `Boolean` 但 DDL 为 `INTEGER`

- **详细设计** 3.1 DDL 中 `is_pinned INTEGER NOT NULL DEFAULT 0`
- **详细设计** 3.4.1 JPA 实体中 `private Boolean isPinned = false;`
- **问题**：SQLite 存储为 INTEGER 0/1，Java 端为 Boolean。Hibernate SQLite Dialect 默认能处理 Boolean↔INTEGER 映射，但需确认 MySQL/PostgreSQL DDL 中 `is_pinned` 的类型定义（MySQL 用 `TINYINT(1)`，PostgreSQL 用 `BOOLEAN`），确保跨数据库一致
- **修正意见**：在 3.2 节 MySQL/PostgreSQL DDL 差异表中已列出布尔值差异（`INTEGER 0/1` | `TINYINT(1)` | `BOOLEAN`），此条仅需确认 Hibernate Dialect 自动映射行为，无需修改文档。编码时注意验证。

---

### S-3. `ScheduledPostPublisher` 中 `markdownService.renderToHtml()` 传入 `post.getId()` 可能导致双向链接解析对未保存文章的影响

- **详细设计** 5.8 中定时发布调度器调用 `post.setContentHtml(markdownService.renderToHtml(post.getContent(), post.getId()))`
- **问题**：此时文章已保存（`post.getId()` 有值），但 `renderToHtml` 内部会调用 `BacklinkService.resolveWikiLinks()`，该方法会查询 `status='public'` 的文章。定时发布时文章状态正在从 `DRAFT` 变为 `PUBLIC`，此时 `post.getId()` 对应的文章还未提交状态变更，BacklinkService 批量查询时不会包含当前文章自身。这在功能上无影响（文章不会引用自身），但需确认事务边界
- **修正意见**：此条为编码时需注意的事务边界问题，文档层面无需修改。建议在编码时确保 `publishScheduledPosts()` 的 `@Transactional` 注解使状态变更和 HTML 渲染在同一事务中完成。

---

### S-4. `SlugUtils.generate()` 中正则 `[^a-z0-9\\u4e00-\\u9fff\\-\\s]` 保留中文字符后又被替换为连字符

- **详细设计** 5.5 中 Slug 生成算法：
  ```java
  .replaceAll("[^a-z0-9\\u4e00-\\u9fff\\-\\s]", "")  // 保留中文
  .replaceAll("[\\u4e00-\\u9fff]", "-")              // 再把中文替换为连字符
  ```
- **问题**：第一步保留中文字符（过滤掉其他特殊字符），第二步又将中文替换为连字符。这两步可以合并为一步：直接 `replaceAll("[^a-z0-9\\-\\s]", "")`，效果等价。当前写法虽然功能正确，但逻辑冗余，可能让开发者困惑为什么先保留再删除
- **修正意见**：建议简化为单步正则，或保留当前实现但添加注释说明两步分离的原因（如未来 v1.1+ 可能引入拼音转换时需要先提取中文）。此为代码风格建议，不影响功能。
